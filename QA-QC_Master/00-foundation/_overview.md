## 1. QA vs QC — ต่างกันอย่างไร?

| หัวข้อ          | QA (Quality Assurance)                        | QC (Quality Control)                      |
| --------------- | --------------------------------------------- | ----------------------------------------- |
| **เป้าหมาย**    | ป้องกันไม่ให้ bug เกิด                        | ตรวจจับ bug ที่เกิดแล้ว                   |
| **แนวคิด**      | Process-oriented                              | Product-oriented                          |
| **ทำเมื่อไหร่** | ตลอดทั้ง development cycle                    | หลัง code เสร็จ ก่อน release              |
| **ตัวอย่าง**    | Code review, coding standards, CI/CD pipeline | Unit testing, E2E testing, manual testing |
| **คนรับผิดชอบ** | ทั้งทีม (everyone owns quality)               | QA Engineer, Tester                       |
| **ผลลัพธ์**     | กระบวนการที่ดีขึ้น                            | ผลิตภัณฑ์ที่ดีขึ้น                        |

### สรุปง่ายๆ

- **QA = ป้องกัน** — สร้างกระบวนการที่ดีเพื่อลดโอกาสเกิด bug ตั้งแต่แรก
- **QC = ตรวจจับ** — ทดสอบและหาข้อผิดพลาดก่อนที่จะถึงมือ user

ทั้งสองอย่างต้องทำควบคู่กัน ไม่ใช่เลือกทำอย่างใดอย่างหนึ่ง

## 2. ทำไมต้องทำทั้ง QA และ QC?

### Cost of Defects — ยิ่งเจอทีหลัง ยิ่งแพง

```
ค่าใช้จ่ายในการแก้ bug (เปรียบเทียบ):

ขั้นตอน Design       : 1x    ████
ขั้นตอน Development   : 6x    ████████████████████████
ขั้นตอน Testing       : 15x   ████████████████████████████████████████████████████████████
ขั้นตอน Production    : 100x  ████████████████████████████████████████... (ยาวมาก)
```

**ตัวอย่างจริง:**

- Bug ที่เจอตอน code review → แก้ใน 10 นาที
- Bug เดียวกันถ้าหลุดไป production → ต้อง hotfix, rollback, แจ้ง user, เสียชื่อ = เสียเวลา + เงินมากกว่า 100 เท่า

### ประโยชน์ที่ได้

| QA ที่ดี                              | QC ที่ดี               |
| ------------------------------------- | ---------------------- |
| ลดจำนวน bug ตั้งแต่ต้น                | ดัก bug ก่อนถึง user   |
| ทีมทำงานเร็วขึ้น (ไม่ต้องแก้ bug ซ้ำ) | มั่นใจก่อน release     |
| Code base สะอาด maintain ง่าย         | ลด customer complaints |
| Onboard คนใหม่ง่ายขึ้น                | ข้อมูล metrics ชัดเจน  |

## 3. Testing Pyramid — สามเหลี่ยมการทดสอบ

```
              /\
             /  \
            / E2E\           ← น้อย, ช้า, แพง
           / (Few)\            Playwright, Cypress
          /--------\
         /Integration\       ← ปานกลาง
        /  (Medium)   \        Supertest, API tests
       /----------------\
      /    Unit Tests     \  ← มาก, เร็ว, ถูก
     /      (Many)         \   Vitest, Jest
    /________________________\
```

### แต่ละชั้นคืออะไร?

| ชั้น                  | จำนวน              | ความเร็ว     | ค่าใช้จ่าย | ทดสอบอะไร                     |
| --------------------- | ------------------ | ------------ | ---------- | ----------------------------- |
| **Unit Tests**        | มากที่สุด (70-80%) | เร็วมาก (ms) | ต่ำ        | Function เดียว, logic เดียว   |
| **Integration Tests** | ปานกลาง (15-20%)   | ปานกลาง (s)  | ปานกลาง    | หลาย module ทำงานร่วมกัน, API |
| **E2E Tests**         | น้อยที่สุด (5-10%) | ช้า (min)    | สูง        | User flow ทั้งหมดบน browser   |

### กฎง่ายๆ

1. **Unit tests เยอะที่สุด** — ทดสอบทุก function, ทุก edge case
2. **Integration tests ปานกลาง** — ทดสอบว่า module ต่างๆ คุยกันถูก
3. **E2E tests น้อยที่สุด** — ทดสอบเฉพาะ critical user flows เท่านั้น

> ถ้า pyramid กลับหัว (E2E เยอะ, Unit น้อย) = สัญญาณอันตราย
> Test จะช้า, แตกบ่อย, maintain ยาก

## 5. หลักการสำคัญ

### Quality ไม่ใช่ขั้นตอน แต่เป็นวัฒนธรรม

- **Shift Left** — เลื่อนการทดสอบมาทำตั้งแต่เริ่ม ไม่ใช่รอตอนท้าย
- **Automate Everything** — สิ่งที่ทำซ้ำได้ ให้ automation ทำ
- **Fail Fast** — ยิ่งเจอ bug เร็ว ยิ่งแก้ง่ายและถูก
- **Continuous Improvement** — วัดผล → หาจุดอ่อน → ปรับปรุง → วัดผลซ้ำ
- **Everyone Owns Quality** — ไม่ใช่แค่ QA team แต่ทุกคนรับผิดชอบ
